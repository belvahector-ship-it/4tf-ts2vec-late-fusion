## ADR-001 — TS2Vec as External Pinned Dependency

**Status:** Approved

### Context
TS2Vec (Yue et al., AAAI 2022) is the encoder backbone of this research. The original implementation is available at `https://github.com/yuezhihan/ts2vec`. The research objective explicitly requires that TS2Vec be used exactly as described in the original paper, without architectural modification. Three integration strategies were considered:

1. Vendor the source code directly into `src/models/`
2. pip-install from PyPI or GitHub as a live dependency
3. Pin a specific commit of the official repository and wrap it externally

### Decision
**Option 3: Pin a specific commit of the official TS2Vec repository. Implement all experiment logic (training, configuration, evaluation, fusion) in our own wrapper modules. Never modify the original TS2Vec source.**

### Rationale
- **Reproducibility:** A pinned commit guarantees that every researcher cloning this repository uses byte-identical TS2Vec code. A live `pip install git+https://...` dependency can silently change if the upstream maintainer pushes commits.
- **Attribution integrity:** The original source remains unmodified and clearly attributed. Vendoring introduces risk of accidental modification and makes attribution less transparent.
- **Separation of concerns:** Our scientific contribution is the experimental protocol, fusion strategy, and evaluation framework — not the encoder. Keeping TS2Vec external makes this separation explicit and reviewer-verifiable.
- **Audit trail:** Reviewers can verify the exact encoder version used by inspecting the pinned commit hash, which is recorded in `requirements.txt`, `environment.yml`, and every checkpoint artifact.

### Consequences
- `requirements.txt` must include the TS2Vec dependency as `git+https://github.com/yuezhihan/ts2vec.git@{COMMIT_HASH}#egg=ts2vec`
- The exact commit hash must be documented in `README.md`, `DS-01`, and every checkpoint reproducibility bundle
- All wrapper code in `src/models/ts2vec_wrapper.py` must import from the pinned installation, not from vendored files
- If the upstream repository becomes unavailable, a fork at `{author_github}/ts2vec` pinned to the same commit serves as fallback (document in README)
- Any future need to modify TS2Vec behavior must be implemented via subclassing or monkey-patching in the wrapper, never by editing the original source, and must be recorded as a new ADR

## ADR-002 — Independent Per-Branch TS2Vec Training

**Status:** Approved

### Context
The multi-branch architecture uses up to four TS2Vec encoders, one per temporal resolution (15m, 1h, 4h, 1d). Three training protocols were considered for conditions with multiple active branches (e.g., 3TF = {15m, 1h, 4h}):

1. **Independent:** Each branch trained separately on its timeframe. Embeddings concatenated post-hoc.
2. **Simultaneous:** All active branches trained in one forward pass with a shared contrastive loss.
3. **Sequential:** Branches trained one at a time, each frozen before the next begins.

### Decision
**Option 1: Each TS2Vec branch is trained independently on its own timeframe data. The four branch encoders (15m, 1h, 4h, 1d) are trained once and their weights are reused across all conditions that include them.**

### Rationale
- **Variable isolation:** Joint or sequential training would allow gradient information from one timeframe to influence another branch's representation. Independent training guarantees that each 64-dim embedding reflects only its own temporal resolution.
- **Efficiency:** Training 4 branches independently means 4 total training runs regardless of how many conditions (1TF–4TF) are evaluated. Joint training would require re-training for each condition.
- **Consistency with TS2Vec design:** TS2Vec was designed as a self-supervised encoder for a single time series stream. Using it in its intended mode — independently on each timeframe — respects the original paper's assumptions.
- **Fusion is purely architectural:** Since branches are pre-trained independently, the concatenation step is a deterministic post-processing operation, not a training objective. This is consistent with ADR-003 (no learnable fusion).

### Consequences
- Exactly 4 training runs are required: one per timeframe (15m, 1h, 4h, 1d)
- Each trained branch produces a checkpoint saved under `checkpoints/branch_{timeframe}/`
- The 1h branch checkpoint is shared by all four conditions (1TF, 2TF, 3TF, 4TF)
- The 15m branch checkpoint is shared by 2TF, 3TF, and 4TF
- The 4h branch checkpoint is shared by 3TF and 4TF
- The 1d branch checkpoint is used only by 4TF
- Branch reuse must be explicitly documented in the experiment log to ensure reviewers understand that conditions share encoder weights, not separate trained models
- Training hyperparameters (optimizer, lr, batch size, epochs, early stopping) must be identical across all four branch training runs to prevent per-branch confounds

## ADR-003 — Fixed Non-Trainable Random Projection for Dimension Standardization

**Status:** Approved

### Context
After independent branch training, each condition produces a concatenated embedding of different dimensionality: 1TF=64-dim, 2TF=128-dim, 3TF=192-dim, 4TF=256-dim. For HDBSCAN clustering and geometric metric comparison across conditions, four strategies were considered:

1. **Learnable Linear projection** (N×64 → 256) trained jointly or separately
2. **Deterministic zero-padding** to 256-dim
3. **Deterministic tiling/repetition** to 256-dim
4. **Natural dimensions** (evaluate each condition in its own embedding space)
5. **Fixed non-trainable random projection** (seeded Gaussian or orthogonal matrix, frozen) mapping N×64 → 256

### Decision
**Option 5: Apply a fixed, non-trainable, seeded random projection matrix to map all conditions to a common 256-dim embedding space. The projection matrix is generated once using a fixed seed, frozen permanently (requires_grad=False), and applied identically to all conditions.**

### Rationale
- **No learnable parameters:** A trainable projection would introduce an additional optimization objective, creating a confound. The projection must add zero trainable parameters to the system.
- **Avoids zero-padding artifacts:** Zero-padding introduces structural sparsity that could artificially deflate intra-cluster distances in lower-TF conditions, biasing geometric metrics.
- **Avoids tiling redundancy:** Tiling artificially amplifies lower-TF embeddings by repeating information, which could artificially inflate cluster separability.
- **Avoids dimensionality confound of natural dimensions:** Comparing Silhouette/DBI/CH scores across spaces of different sizes (64 vs 128 vs 192 vs 256) conflates representation quality with the curse of dimensionality.
- **Random projection is theoretically grounded:** The Johnson-Lindenstrauss lemma guarantees that random projections approximately preserve pairwise distances. A fixed seeded projection is deterministic and reproducible.
- **Reviewer defensibility:** Fixed random projection is a well-established dimensionality-standardization technique. It is neutral — it neither advantages nor disadvantages any condition.

### Implementation Specification
```
Projection matrix P ∈ R^{256 × (N×64)}
Generated as: torch.randn(256, N*64, generator=torch.Generator().manual_seed(PROJECTION_SEED))
Normalized: each row normalized to unit L2 norm (orthogonal-like behavior)
Frozen: P.requires_grad = False
Applied: z_256 = P @ z_concat  where z_concat ∈ R^{N×64}
Seed: PROJECTION_SEED = 42 (recorded in config and every checkpoint)
```

Note: A separate projection matrix P is generated for each condition (since input dimension N×64 differs). Each P is generated with the same PROJECTION_SEED for reproducibility.

### Consequences
- The projection matrices must be saved in each checkpoint bundle (ADR-010) to guarantee identical projection at inference/evaluation time
- PROJECTION_SEED must be a top-level config parameter, never hardcoded
- The paper's methodology section must describe this as "fixed random projection for dimension standardization" and cite the Johnson-Lindenstrauss justification
- The sensitivity analysis (ADR-004, Stage 2) must also use identical projection matrices to ensure comparability

### Limitation to Document in Paper
A fixed random projection, while theoretically sound, is not a learned compression. The 256-dim representation for lower-TF conditions contains the full information of the original N×64 embedding, linearly mixed. This is an acknowledged limitation and must appear in the paper's Limitations section.

## ADR-004 — Two-Stage HDBSCAN Hyperparameter Protocol

**Status:** Approved

### Context
HDBSCAN requires two key hyperparameters: `min_cluster_size` and `min_samples`. Different hyperparameter choices can produce qualitatively different clustering results. In a controlled experiment comparing conditions (1TF–4TF), allowing each condition to independently select its own hyperparameters introduces a confound: observed differences in cluster quality could reflect hyperparameter differences rather than embedding quality differences. Two strategies were considered:

1. **Fixed from 1TF:** Grid search on 1TF only, lock parameters for all conditions
2. **Per-condition grid search:** Each condition independently selects best hyperparameters

### Decision
**A two-stage protocol:**

**Stage 1 (Primary Experiment):** Grid search is performed on the 1TF baseline only. The hyperparameters that maximize Silhouette Score on 1TF are locked and applied unchanged to 2TF, 3TF, and 4TF. This is the primary result reported in the ablation table and used to answer RQ1 and RQ2.

**Stage 2 (Sensitivity Analysis):** An independent per-condition grid search is performed after the primary experiment. Results are reported in a supplementary table. If conclusions differ between Stage 1 and Stage 2, this is discussed explicitly as a robustness finding.

### Rationale
- **Experimental validity:** Stage 1 isolates temporal resolution as the sole independent variable. Any observed metric differences across conditions must be attributable to representation quality, not to hyperparameter shopping.
- **Transparency:** Stage 2 provides an honest robustness check. If the conclusions hold under per-condition grid search, the finding is stronger. If they differ, this is a meaningful scientific observation about the interaction between representation dimensionality and HDBSCAN sensitivity.
- **Reviewer expectations:** SINTA 1 reviewers may ask whether results are sensitive to clustering hyperparameters. Stage 2 directly pre-empts this concern.

### Grid Search Space
```
min_cluster_size : [50, 100, 200]
min_samples      : [5, 10, 20]
metric           : 'euclidean' (fixed)
max_k            : 8 (hard constraint, per ADR-018)
Selection criterion: highest Silhouette Score among configurations producing 2 ≤ k ≤ 8 clusters
```

### Consequences
- Stage 1 must be completed and hyperparameters locked before any multi-TF condition is clustered
- All selected hyperparameters (per condition, per seed, per stage) must be saved in the experiment artifact (metrics CSV and run log)
- Stage 2 is labeled "sensitivity analysis" in all outputs to prevent confusion with the primary result
- If Stage 1 hyperparameters produce fewer than 2 non-noise clusters for any condition, a fallback procedure must be documented: expand grid search to `min_cluster_size ∈ [20, 50, 100, 200]` and re-lock from 1TF

## ADR-005 — Cumulative 1h-Anchored Condition Definition with Per-Timeframe Baselines

**Status:** Approved

### Context
The experiment tests four multi-resolution conditions. The research question is whether adding temporal resolutions improves representation quality. Two approaches to defining conditions were considered:

1. **Cumulative anchored on 1h:** 1TF={1h}, 2TF={15m,1h}, 3TF={15m,1h,4h}, 4TF={15m,1h,4h,1d}
2. **All combinatorial pairs for 2TF:** 6 additional conditions testing every possible pair

Additionally, Option 3 from clarification B3 adds individual per-timeframe baselines: {15m}, {1h}, {4h}, {1d} as four separate 1TF runs.

### Decision
**Cumulative 1h-anchored conditions (1TF–4TF) as the primary experiment, plus four individual per-timeframe baselines ({15m}, {1h}, {4h}, {1d}) as secondary baselines.**

The 1h branch serves as the fixed reference. Resolutions are added incrementally in order of increasing coarseness: fine-to-coarse addition (15m, then 4h, then 1d) around the 1h anchor.

### Rationale
- **Interpretability:** Cumulative design allows a clean narrative: "what happens as we progressively add coarser context to the 1h anchor?"
- **Alignment with RQ2:** "How does the number of temporal resolutions influence quality?" is directly answered by the cumulative progression.
- **Per-timeframe baselines provide individual contribution evidence:** Knowing that {15m} alone vs {1h} alone produce different quality helps interpret why 2TF behaves as it does.
- **Combinatorial design is out of scope for this paper:** 6 conditions for 2TF alone would require 6×5 = 30 additional runs (30 seeds) and would shift the paper's focus from controlled ablation to combinatorial search.

### Consequences
- Total branch training runs: 4 (one per timeframe, independent per ADR-002)
- Total experimental conditions: 7 (4 per-timeframe baselines + 3 cumulative conditions)
- Total experiment runs: 7 conditions × 5 seeds = 35 runs
- Per-timeframe baselines ({15m}, {4h}, {1d}) are secondary results and reported separately from the primary ablation table
- The primary ablation table covers only the 4 cumulative conditions (1TF–4TF) with 1TF={1h} as baseline
- All 7 conditions use the same locked HDBSCAN hyperparameters from Stage 1 (selected on 1TF={1h})

## ADR-006 — W=48 Sliding Window with Stride=1 Fixed Across All Conditions

**Status:** Approved

### Context
TS2Vec requires a sequence-length input. The blueprint specifies W=48 for z-score normalization but does not specify stride for the sliding window. Stride controls the overlap between consecutive windows and directly affects dataset size and computational cost.

### Decision
**W=48 (48 consecutive 1h-anchor candles = 48 hours), stride=1 (dense sliding window), fixed and identical across all conditions and all timeframes.**

### Rationale
- **Maximum training samples:** Stride=1 produces approximately 26,280 − 48 + 1 ≈ 26,233 training windows from the 3-year training period, maximizing the amount of self-supervised signal available to each branch encoder.
- **Alignment with TS2Vec design philosophy:** TS2Vec uses random cropping internally during training. A dense sliding window provides the full temporal coverage from which random crops are drawn.
- **Controlled confound:** Fixing W and stride across all conditions ensures that dataset size and temporal coverage are identical for each condition, preventing any condition from having more training signal than another.
- **Z-score normalization alignment:** The blueprint's W=48 for z-score normalization aligns exactly with the window size, meaning each input window is normalized over its own 48-candle context — a clean, self-contained operation.

### Consequences
- The sliding window is applied independently to each timeframe's feature matrix before feeding to its branch encoder
- For the 15m timeframe: W=48 means 48 fifteen-minute candles (12 hours of data) per window
- For the 4h timeframe: W=48 means 48 four-hour candles (8 days of data) per window
- For the 1d timeframe: W=48 means 48 daily candles (48 days of data) per window
- Note: each timeframe branch sees its own natural temporal resolution — the 1h anchor alignment (ADR-017) means all branches have synchronized timestamps at the 1h level, but each branch's W=48 window represents a different real-world time span
- This is a feature, not a bug: it is precisely the multi-scale temporal context that the research hypothesizes will improve representations
- Any future experiment varying W must be registered as a new ablation study, not a modification to this protocol

## ADR-007 — Device-Agnostic Execution with GPU-First Automatic Fallback

**Status:** Approved

### Context
The repository must run on local machines (CUDA GPU), Google Colab (CUDA GPU), Kaggle (CUDA GPU), and reviewer environments (CPU only). Two strategies were considered:

1. Separate CPU and GPU code paths
2. Single device-agnostic code path with automatic detection

### Decision
**Single device-agnostic code path. Device is detected once at startup and passed through the entire pipeline via a `device` configuration object. No code path distinction between CPU and GPU.**

### Implementation Specification
```python
import torch

def get_device() -> torch.device:
    """Detect and return the best available compute device."""
    if torch.cuda.is_available():
        return torch.device("cuda")
    return torch.device("cpu")
```

All tensors, models, and data loaders must use `tensor.to(device)` consistently. No `torch.cuda.*` calls that bypass device abstraction are permitted.

### Consequences
- Checkpoint files saved on GPU must be loadable on CPU: use `torch.load(..., map_location='cpu')` in all loading utilities
- Random seed behavior differs slightly between CPU and GPU for some operations. This must be documented and tested
- Performance on CPU will be significantly slower. The README must document expected runtime on both CPU and GPU
- Mixed precision training (`torch.cuda.amp`) is permitted as an optional optimization flag in config but must gracefully disable on CPU

## ADR-009 — Parquet + NumPy Dual-Format Dataset Artifact Policy

**Status:** Approved

### Context
Processed datasets need to be stored efficiently and in formats suitable for different downstream uses: tabular analysis and direct DataLoader consumption.

### Decision
**Two artifact formats per processed dataset stage:**
1. **Apache Parquet** for all tabular data (raw OHLCV, aligned 1h master, train/test splits with timestamps and feature columns)
2. **NumPy .npy** for windowed feature tensors ready for direct DataLoader consumption (shape: `[N_windows, W, F]` where W=48, F=7)

### Rationale
- Parquet is columnar, compressed, and preserves dtypes (including datetime). It is the standard format for financial time series in production and research.
- NumPy .npy allows zero-copy loading into PyTorch DataLoaders via `torch.from_numpy()`. No re-computation of windows is needed at training time.
- Separating the formats separates concerns: Parquet is for analysis and joining; NumPy is for training.

### Consequences
- Window generation must be a deterministic, reproducible operation (same config → identical .npy)
- .npy files are large and must be excluded from git tracking (`.gitignore`). Parquet files for processed data are also excluded; only raw download metadata is committed.
- A data manifest file (`data/manifest.json`) must record file sizes, row counts, shape, and SHA-256 checksums of all generated artifacts

## ADR-010 — Full Reproducibility Bundle Checkpoint Policy

**Status:** Approved

### Context
Model checkpoints are needed to resume training, evaluate models, and allow other researchers to reproduce results without retraining. Minimal checkpoints (weights only) are insufficient for a public research repository.

### Decision
**Every checkpoint is a complete reproducibility bundle containing:**

```python
{
    # Model state
    "model_state_dict": {...},          # Branch encoder weights
    "branch_timeframe": "1h",           # Which timeframe this branch encodes
    
    # Training state (for resume)
    "optimizer_state_dict": {...},
    "scheduler_state_dict": {...},       # None if no scheduler used
    "epoch": int,
    "global_step": int,
    "best_metric": float,               # Best validation loss seen
    "best_epoch": int,
    
    # Experiment identity
    "seed": int,                        # Random seed for this run
    "condition": "2TF",                 # Experimental condition
    "config_snapshot": {...},           # Full YAML config as dict
    
    # Reproducibility metadata
    "timestamp": "ISO-8601 string",
    "git_commit_hash": "str | None",    # None if not in git repo
    "python_version": "3.11.x",
    "torch_version": "x.x.x",
    "numpy_version": "x.x.x",
    "ts2vec_commit": "sha1 hash",       # Pinned TS2Vec commit
    
    # Projection matrix (ADR-003)
    "projection_matrix": torch.Tensor,  # Fixed random projection P
    "projection_seed": int,
    
    # Training metrics history
    "train_loss_history": [float, ...],
    "epoch_times": [float, ...],        # Seconds per epoch
}
```

Two checkpoint files are maintained per training run:
- `best_model.pt` — checkpoint at epoch with lowest training loss
- `latest_model.pt` — checkpoint at most recent completed epoch (for resume)

### Consequences
- Checkpoint files will be larger than weights-only saves (~5–50 MB depending on model size)
- Checkpoints must not be committed to git. They are uploaded separately (e.g., Zenodo, Google Drive, or Hugging Face Hub) and linked from README
- A checkpoint loading utility must handle version mismatches gracefully and warn if metadata fields are missing (backward compatibility for future researchers)

## ADR-011 — NumPy + Parquet Dual-Format Embedding and Cluster Artifact Policy

**Status:** Approved

### Context
After training and inference, each experimental run produces embeddings (256-dim vectors per observation) and cluster labels. These need to support both downstream tensor operations and economic validity analysis.

### Decision
**Two artifact formats per experiment run:**

1. **NumPy .npy** files:
   - `embeddings_train.npy` — shape `[N_train, 256]`
   - `embeddings_test.npy` — shape `[N_test, 256]`
   - `cluster_labels_train.npy` — shape `[N_train]`, dtype int, -1 for noise
   - `cluster_labels_test.npy` — shape `[N_test]`, dtype int, -1 for noise

2. **Apache Parquet** files:
   - `embeddings_train.parquet` — timestamps + OHLCV columns + 256 embedding dimensions + cluster label + is_noise flag
   - `embeddings_test.parquet` — same schema

### Rationale
- .npy files enable fast metric computation (Silhouette, DBI, CH) directly in NumPy/scikit-learn
- Parquet files join embeddings with timestamps and OHLCV for economic validity analysis (return distributions per cluster) and timeline visualization (Figure 3)
- The Parquet schema allows SQL-style queries: "show me all 1h candles in cluster 2 during 2023"

### Consequences
- HDBSCAN noise points (label=-1) are preserved in both formats and must be explicitly excluded from geometric metric computation
- Noise percentage must be reported as an additional metric per run
- The Parquet schema is fixed and documented in DS-03 (Module Interface Contracts)

## ADR-013 — Late Fusion via Deterministic Concatenation (No Learnable Fusion Parameters)

**Status:** Approved

### Context
After independent branch training (ADR-002), embeddings from active branches must be combined into a single representation. Several fusion strategies were considered including learnable linear projection, early fusion, and cross-attention.

### Decision
**Late fusion via deterministic concatenation only. Active branch embeddings are concatenated along the feature dimension. No learnable parameters participate in the fusion step.**

```
1TF: concat([e_1h])                         → 64-dim
2TF: concat([e_15m, e_1h])                  → 128-dim
3TF: concat([e_15m, e_1h, e_4h])            → 192-dim
4TF: concat([e_15m, e_1h, e_4h, e_1d])     → 256-dim
```

Then apply fixed random projection (ADR-003) → 256-dim for all conditions.

The concatenation order is fixed: fine-to-coarse (15m, 1h, 4h, 1d). This order is consistent across all conditions and must not vary between runs.

### Rationale
- Learnable fusion would introduce trainable parameters that could absorb information in ways that confound the temporal resolution variable
- Early fusion would prevent each branch from developing resolution-specific representations
- Cross-attention is a more expressive architecture and is explicitly deferred to paper 2 (blueprint Section 19)
- Deterministic concatenation is the simplest possible fusion consistent with the research framing as a "controlled empirical study"

### Consequences
- The concatenation operation has zero hyperparameters and is not logged separately
- Branch embedding order (15m, 1h, 4h, 1d) is fixed and must be enforced programmatically via an ordered list, not a set or dict
- Any future study varying the concatenation order must be recorded as a new ADR and treated as a separate ablation

## ADR-014 — Walk-Forward Temporal Split (Train 2020–2022, Test 2023)

**Status:** Approved (from blueprint D-12)

### Context
Financial time series must not be split randomly. Random splits cause data leakage because future price information would appear in the training set.

### Decision
**Walk-forward temporal split:**
- **Train:** 2020-01-01 00:00:00 UTC → 2022-12-31 23:00:00 UTC
- **Test:** 2023-01-01 00:00:00 UTC → 2023-12-31 23:00:00 UTC

No shuffling. No stratification. No overlap.

### Rationale
- Temporal ordering is preserved, preventing future leakage
- 2023 represents a structurally different market regime (post-FTX collapse recovery, Bitcoin ETF anticipation) providing a realistic external validity test
- The 3:1 ratio provides sufficient training data for self-supervised learning while maintaining a full-year evaluation period

### Consequences
- The split boundary (2022-12-31 23:00:00 UTC) must be enforced programmatically and verified by assertion in the preprocessing pipeline
- All z-score normalization statistics (mean, std per window) must be computed from training data only. Test set windows are normalized using statistics computed from their own window (per-window z-score), which is safe since no training statistics are used at test time.
- HDBSCAN is fit on training embeddings only. Test embeddings are assigned to the nearest training cluster using HDBSCAN's approximate prediction functionality.

## ADR-015 — Seven OHLCV-Derived Features Without Technical Indicators

**Status:** Approved (from blueprint D-05, Section 8)

### Context
Feature engineering choices can implicitly introduce temporal smoothing at different scales depending on the timeframe, creating a hidden confound.

### Decision
**Exactly seven features, all derived directly from OHLCV data, with no technical indicators:**

| # | Feature | Formula |
|---|---|---|
| 1 | `open_return` | (open − prev_close) / prev_close |
| 2 | `high_return` | (high − open) / open |
| 3 | `low_return` | (low − open) / open |
| 4 | `close_return` | (close − open) / open |
| 5 | `volume_zscore` | (volume − rolling_mean₂₀) / rolling_std₂₀ |
| 6 | `hl_range` | (high − low) / open |
| 7 | `body_ratio` | abs(close − open) / (high − low + 1e-8) |

### Rationale
Technical indicators (RSI, MACD, Bollinger Bands, etc.) embed period parameters (e.g., RSI-14 means 14 candles). Applying RSI-14 to a 15m series vs a 1d series captures fundamentally different time spans. This would inject implicit temporal information through the feature engineering rather than through the architecture, creating an uncontrolled confound.

### Consequences
- Feature computation must be implemented identically for all four timeframes
- `volume_zscore` uses a rolling window of 20 candles in the timeframe's native resolution (20 × 15m for the 15m branch, etc.). This is the only feature with a look-back window and must be carefully documented as a source of minor, explicit, and quantifiable temporal smoothing
- The first 20 candles of each timeframe series will have NaN volume_zscore and must be handled explicitly (forward-fill or drop, documented in DS-02)
- `body_ratio` uses +1e-8 epsilon in the denominator to prevent division by zero on doji candles

## ADR-016 — Per-Window Z-Score Normalization

**Status:** Approved (from blueprint Section 8)

### Context
Financial time series exhibit non-stationarity (trend, regime changes, volatility clustering). Global normalization would allow the model to learn absolute price level information rather than relative patterns.

### Decision
**Per-window z-score normalization applied to each W=48 window independently, after feature computation:**

```
For each window w of shape [48, 7]:
    mean = w.mean(axis=0)   # shape [7]
    std  = w.std(axis=0)    # shape [7]
    w_normalized = (w - mean) / (std + 1e-8)
```

Normalization is applied per-window, per-feature (column-wise within the window).

### Rationale
- Per-window normalization removes the absolute level of each feature within the window, forcing the encoder to learn the shape and dynamics of the window rather than its absolute value
- This is consistent with TS2Vec's design intent: learning contextual, scale-invariant representations
- No training-set statistics are required for normalization, eliminating any risk of leakage from the normalization step

### Consequences
- Normalization is applied during window generation (before DataLoader), not inside the model
- Windows with near-zero standard deviation (e.g., a flat sideways market) are handled by the +1e-8 epsilon
- The normalization parameters (mean, std per window) are not saved — they are recomputed on the fly during window generation, which is deterministic given the same input data

## ADR-017 — Forward-Fill Upsampling for 4h and 1d Timeframes

**Status:** Approved (from blueprint Section 8)

### Context
The 1h anchor requires all timeframes to provide a value at every 1h timestamp. Lower-frequency timeframes (4h, 1d) must be upsampled to match.

### Decision
**Forward-fill (last observation carried forward) for upsampling 4h and 1d to 1h resolution:**
- One 4h candle is broadcast to the next 4 consecutive 1h timestamps
- One 1d candle is broadcast to the next 24 consecutive 1h timestamps

For downsampling 15m to 1h:
- Four 15m candles are aggregated: open=first, high=max, low=min, close=last, volume=sum

### Rationale
- Forward-fill is the only upsampling strategy that does not use future information. Interpolation between a current and future 4h candle would constitute look-ahead leakage.
- The aggregation rules for 15m→1h are standard OHLCV aggregation conventions.

### Consequences
- A mandatory unit test must verify that forward-fill does not introduce look-ahead: at timestamp T, the 4h branch value must reflect only information available at or before T
- The first partial 4h period and first partial 1d period at the dataset start boundary must be handled explicitly (documented in DS-02)
- Forward-fill creates artificial autocorrelation in the 4h and 1d branch features (the same value repeated for 4 or 24 consecutive hours). This is documented as a limitation in the paper (blueprint OI-4)

## ADR-018 — HDBSCAN Maximum Cluster Constraint (k ≤ 8)

**Status:** Approved (from blueprint Section 9)

### Context
HDBSCAN can in principle discover any number of clusters. Without a maximum, configurations with very small `min_cluster_size` might produce dozens of micro-clusters that are not interpretable as market states.

### Decision
**Hard constraint: configurations producing more than 8 clusters are excluded from hyperparameter selection. Selection criterion: highest Silhouette Score among configurations producing 2 ≤ k ≤ 8 non-noise clusters.**

### Rationale
- Market state interpretation requires human-readable labels (Bull, Bear, Sideways, High-Volatility). Beyond ~6–8 states, distinctions become ambiguous.
- The constraint is consistent with domain knowledge: most market regime literature identifies 2–5 distinct states.
- The lower bound of k ≥ 2 excludes degenerate solutions where all points are assigned to a single cluster.

### Consequences
- If no HDBSCAN configuration in the grid produces 2 ≤ k ≤ 8 clusters, the grid must be expanded and this must be logged as a warning
- Noise points (label=-1) are not counted toward k
- k is reported as a metric per run alongside geometric metrics

## ADR-019 — Five-Seed Statistical Protocol with Wilcoxon Signed-Rank Test

**Status:** Approved (from blueprint Section 15)

### Context
Single-run results in machine learning are not statistically reliable. A multi-seed protocol with formal statistical testing is required for SINTA 1 publication.

### Decision
**Five seeds [42, 123, 456, 789, 1024] applied to all randomness sources (Python, NumPy, PyTorch, CUDA). Results reported as mean ± std across seeds. Primary statistical test: Wilcoxon signed-rank test comparing the best multi-TF condition vs 1TF baseline across 5 paired seed runs. Significance threshold: p < 0.05.**

### Rationale
- Wilcoxon signed-rank is appropriate for small samples (n=5) with unknown distribution
- Paired test is correct because the same 5 seeds are used for all conditions — the comparison is within-seed, removing seed-to-seed variance as a confound
- 5 seeds is the minimum for a publishable statistical argument; more would strengthen the claim but significantly increase compute cost

### Consequences
- Total experiment runs: 7 TS2Vec conditions (4 primary cumulative + 3 secondary per-timeframe baselines, per ADR-005) × 5 seeds = 35 runs, plus 2 external baselines (HMM, KM-PCA, per DS-03 §4) × 5 seeds = 10 additional runs. **Total: 45 runs.** *(Corrected in v1.1 — see document changelog. This figure now agrees with ADR-005 "Consequences" and with DS-03 v1.2 §6.)*
- The Wilcoxon test is performed on Silhouette Score as the primary metric
- p-values must be reported to three decimal places in the paper
- Effect size (rank-biserial correlation r) should be reported alongside p-value

## ADR-021 — Accept TS2Vec's Internal Optimizer Defaults (No Monkey-Patch/Subclass)

**Status:** Approved

**Relates to:** ADR-001 (TS2Vec as External Pinned Dependency), ADR-002 (Independent Per-Branch TS2Vec Training), ADR-010 (Full Reproducibility Bundle Checkpoint Policy). *(Consolidated into this index in v1.2; originally created as a standalone addendum during the M7 implementation session.)*

### Context
M7 (TS2Vec Wrapper) implementation revealed that the pinned TS2Vec constructs its `torch.optim.AdamW` optimizer **inside** `TS2Vec.fit()` as a local variable, passing only `lr`:

```python
optimizer = torch.optim.AdamW(self._net.parameters(), lr=self.lr)
```

Two consequences follow, both structural to the upstream code:

1. **`weight_decay` is not configurable.** Upstream uses the torch default `weight_decay = 0.01`; our `base.yaml training.weight_decay: 0.0001` is never read by the encoder's optimizer.
2. **The optimizer object is not exposed**, so no `optimizer_state_dict` can be captured for a true optimizer-state resume (ADR-010 lists this field).

Two options were considered: (a) monkey-patch or subclass TS2Vec to inject `weight_decay = 1e-4` and expose the optimizer; (b) accept TS2Vec's internal defaults, leaving the third-party code untouched.

### Decision
**Option (b): accept TS2Vec's internal optimizer defaults. Do not monkey-patch or subclass the TS2Vec library. Use TS2Vec exactly as pinned.**

### Rationale
- `weight_decay` is an **internal parameter of TS2Vec** (a third-party, vendored/pinned dependency), not part of this research's contribution. The novelty is the Cross-Timeframe Attention and Late Fusion architecture built *on top of* TS2Vec — not the internal tuning of TS2Vec.
- The project is deliberately **neutral toward upstream TS2Vec**, consistent with ADR-001 (pinned dependency; vendoring is read-reference only). Monkey-patching the optimizer would violate the spirit of ADR-001 and add new complexity and risk to already-validated third-party code.
- `base.yaml training.weight_decay: 0.0001` is **not removed** — it remains as documentation of the design-intended value. The wrapper explicitly records that this value is not actually used (superseded by the library default `0.01`).
- This is documented honestly as a **limitation** in the paper's methodology.
- Related consequence: `optimizer_state_dict` remains `None`; **weight-level resume** (re-`fit` from saved encoder weights), not true optimizer-state resume, is accepted for the same neutrality-toward-upstream reason.

### Consequences
- `src/models/ts2vec_wrapper.py` records `config_weight_decay` (0.0001) and `effective_weight_decay` (0.01, read from the live torch AdamW default) in every checkpoint bundle, and logs a WARNING at construction when they differ.
- ADR-010's `optimizer_state_dict` field is written as `None` for all branch checkpoints; M8's `load_or_train` resume is weight-level, not optimizer-state-level.
- No change to `configs/base.yaml`. The paper's Limitations/Methodology section must state this explicitly.
- If a future finding requires honoring `weight_decay = 1e-4`, that would be a new ADR superseding this one (with the monkey-patch/subclass path and its ADR-001 implications) — out of scope now.

## ADR-022 — `no_excessive_single_gap` Is a Warning, Not a Hard Gate

**Status:** Approved

**Relates to / amends:** DS-04 V-DATA-001 (M2 Data Validation pass criteria), DS-02 Stage 1. *(Consolidated into this index in v1.2; originally created as a standalone addendum on the first real-data pipeline run.)*

### Context
DS-04 V-DATA-001 required, among the M2 pass criteria: *"No single gap exceeds one candle duration by more than one unit"* (≤ 2× the candle duration). On the **first run of M1 with real Binance data**, this failed for 15m and 1h because **Binance itself had multi-hour outages** during 2020–2023:

| TF | single gaps > 2 candles | largest gap |
|----|-------------------------|-------------|
| 15m | 15 | 24 candles (6h) @ 2020-02-19 |
| 1h | 8 | 6 candles (6h) @ 2020-02-19 |
| 4h | 0 | 2 candles |
| 1d | 0 | 1 candle |

The largest gap (2020-02-19) is the well-known Binance outage that day; the others (2021-04-25, 2021-08-13, 2020-12-21, 2020-06-28) are likewise real exchange downtime. No duplicates; the aggregate gap ratio is only ~0.11% (far within the 5% tolerance). The strict single-gap criterion is **unsatisfiable for real 15m/1h Binance data** and only holds for 4h/1d.

### Decision
**`no_excessive_single_gap` is downgraded from a hard failure to a logged WARNING.** It is still computed, recorded in the `ValidationReport`, and logged at WARNING level; it no longer aborts M2. The **aggregate `gap_ratio` (5%) remains the hard data-quality gate — unchanged.**

### Rationale
- Multi-hour exchange downtime is a genuine property of the market being studied, not a data defect. Hard-failing on legitimate downtime is counter-productive for research representing real market conditions.
- The aggregate gap ratio (0.11% ≪ 5%) is the statistically meaningful completeness guard.
- M3 (Temporal Alignment) forward-fills / reindexes to a complete 1h grid by design (see ADR-023), so these gaps are resolved before modeling; the strict raw-level single-gap check adds no protection M3 does not already provide.
- Auditability is preserved: the gap is logged as a WARNING and rendered `[WARN]` in the report summary.

### Consequences
- `CheckResult` gains a `severity` field (`"error"` default, `"warning"`). `ValidationReport.passed` considers only error-severity checks; a `warnings` property lists failed warning-severity checks; `summary()` renders `[WARN]`.
- `no_excessive_single_gap` is appended with `severity="warning"` and, on failure, emits an explicit WARNING with the timeframe, gap size, and aggregate ratio.
- DS-04 V-DATA-001 was reworded: the single-gap item becomes "logged as a warning; does not abort"; the aggregate 5% gap-ratio remains the pass/fail gate.
- Regression tests confirm a large single gap with aggregate ratio < 5% passes but is recorded as a warning, while aggregate ratio > 5% still hard-fails. No change to M3–M6 or any research decision.

## ADR-023 — M3 Reindexes to a Complete Hourly Grid (Outage-Hole Fill)

**Status:** Approved

**Relates to:** DS-02 Stage 2 (Temporal Alignment), DS-04 V-DATA-003 / V-LEAK-001, ADR-022. *(Consolidated into this index in v1.2; originally created as a standalone addendum on the first real-data pipeline run.)*

### Context
M3 aligns all four timeframes to the 1h anchor. `build_master` originally set the target index to the **actual** 1h candle timestamps. With clean data that is the full 35,064-hour grid, but real Binance data is missing **32 hours** (exchange outages), so the master came out to **35,032 rows** and failed the schema check. Anchoring on actual candles would shorten the master and shift every downstream count the pipeline depends on — including the LC-4-audited **N_test_windows = 8,760 (exact)**, M4's 35,045-row feature matrix, and the split sizes. The M3 docstring intent was a complete grid ("every 1h timestamp … has exactly one row"), which only held by accident when the data had no gaps.

### Decision
**M3 reindexes the 1h anchor (and the 15m-aggregated series) to a complete, gap-free hourly grid** spanning `[min, max]` of the observed 1h timestamps. For every hour with no source candle (an outage hole): **prices (O/H/L/C) are last-observation-carried-forward (LOCF)** — from a prior candle only, never a future one (no look-ahead) — and **volume is set to 0.0** (no trading occurred). The 4h/1d columns are unchanged (already forward-filled onto the target index; their real candle covering an outage hour retains its real volume).

### Rationale
- A complete hourly grid is the **factual representation of elapsed time**, independent of exchange up/down status; deleting outage hours would distort the temporal structure every downstream module depends on.
- **LOCF prices** are the standard "no trading" assumption, with **no look-ahead bias**.
- **volume = 0** (not forward-filled) is factually correct: no volume traded during the outage. Forward-filling volume would fabricate trading that did not happen.
- Consistent with ADR-022 (large gaps accepted as legitimate market conditions, handled here technically).
- **Preserves the LC-4-audited counts** — the base grid is the full 35,064 hours the LC-4 audit assumed; N_test_windows=8,760 and all downstream exact figures remain valid, **proven** by re-running M4/M5/M6 on the reindexed real data (not assumed).

### Consequences
- `TemporalAligner._reindex_highfreq_to_grid` performs the LOCF-price / volume-0 fill; `build_master` builds the target index from a complete `pd.date_range(min, max, freq="1h")`.
- New `verify_grid_completeness(master)` asserts every step is exactly 1h; checked in `build_and_verify_master` alongside the schema and V-LEAK-001 look-ahead checks.
- V-LEAK-001 (no look-ahead) continues to pass on real data (8/8 sampled timestamps); the LOCF fill uses only prior candles.
- Regression tests: a punched 1h/15m hole is reindexed back, prices are LOCF, volume is 0, the grid is complete, and the end-to-end leakage check still passes. Outage-window values differ slightly from the clean-data assumption, but **row counts and all LC-4 exact figures are unchanged**. No research decision changes; DS-02 Stage 2 gains this gap-fill rule.

## ADR-024 — KM-PCA `n_components` Clamped to Available Feature Dimension

**Status:** Approved

**Relates to:** DS-03 §4 (External Baselines), IMP-01 M10.5, base.yaml `external_baselines.km_pca`. *(Consolidated into this index in v1.2; originally created as a standalone addendum during the M10.5 implementation session.)*

### Context
The KM-PCA external baseline specification contained an internal contradiction present since the earliest design draft, surfaced when M10.5 implementation began. DS-03 §4, base.yaml, and IMP-01 all say KM-PCA uses **"7 features, 1h resolution"** *and* `PCA(n_components=10)`. **Mathematically impossible:** PCA cannot produce more components than `min(n_samples, n_features)`; with 7-feature input, `PCA(n_components=10)` raises a `ValueError`. The value `10` is almost certainly a residue from an early draft that imagined a higher-dimensional feature space, never reconciled after the final "7 features, 1 timeframe" decision was locked for all baselines (including the 1TF/BL-1h condition KM-PCA is compared against). No document justifies `10` explicitly.

### Decision
**Clamp:** `n_components = min(configured_pca_components, n_features_available)`. For the 7-feature/1h input this yields **`PCA(n_components=7)`** — a full-rank orthogonal projection (whitening/decorrelation), NOT dimensionality reduction. Implemented as an **explicit clamp** computed at fit time, not a silent override. Two rejected alternatives: `PCA=5` (an arbitrary number with no document basis); using all 28 features so `PCA(10)` is literal (breaks the documented "7 features, 1h resolution" and gives KM-PCA an unfair information advantage over 1TF/BL-1h).

### Rationale
- **Documentation consistency:** "7 features, 1h resolution" is locked in DS-03 §4, base.yaml, IMP-01 M10.5, and PROPOSAL §3.1/§3.6. No document ever specified PCA components exceeding 7.
- **Fair comparison (INV-001, single independent variable):** KM-PCA must have information access **identical** to the 1TF/BL-1h condition — not more.
- **No invented parameters:** clamping introduces no new undocumented number.
- **Methodologically valid:** `n_components = n_features` is a legitimate boundary case — PCA functions as covariance-structure standardization (whitening/decorrelation) before clustering, not compression. No information is discarded; the data is only rotated to its principal-component basis.

### Consequences
- `KMeansPCABaseline` computes `effective_pca_components = min(pca_components, n_features)` (7 for the 1h/7-feature input) and exposes it for audit; the PCA `explained_variance_ratio_` is saved with each run.
- For this input PCA is full-rank whitening, **not** dimensionality reduction — documented as a **limitation of the external baseline** (to appear in the paper's Limitations section), not a bug.
- `base.yaml`'s `km_pca.pca_components: 10` is reinterpreted as an **upper bound / ceiling**, with a clarifying comment; the effective value is computed at runtime via the clamp.
- No research decision changes; DS-03 §4 gains a footnote at its next revision. A regression test asserts effective components = 7 (not 10) and that no `ValueError` is raised on the 7-feature input.
