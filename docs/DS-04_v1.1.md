## Purpose

This document defines the validation activities required to confirm that the implementation faithfully follows the research protocol specified in DS-03.

Every validation included here protects research validity. If a validation activity only improves software quality without affecting scientific validity, it does not belong in this document.

---

## 1. Validation Principles

### 1.1 Purpose of Validation

The implementation of this research involves multiple pipeline stages, each of which can introduce errors that silently invalidate the experimental comparison. The purpose of validation is not software quality assurance — it is scientific integrity assurance.

A passing implementation is one where a reviewer or independent researcher could inspect the validation results and conclude that the experimental protocol was followed correctly.

### 1.2 Relationship with DS-03 Research Invariants

Each Research Invariant defined in DS-03 Section 1 (INV-001 through INV-008) has a corresponding validation item in Section 4 of this document. An implementation that cannot pass its invariant validation is considered non-compliant with the research protocol, regardless of whether it produces plausible results.

All other validations in this document are derived from the leakage checkpoints defined in DS-02, the experimental conditions defined in DS-03 Section 4, and the statistical protocol defined in DS-03 Section 6.

---

## 2. Validation Categories

**Data Validation** confirms that the input data satisfies the integrity conditions assumed by the research protocol. Errors here propagate silently through all downstream stages.

**Temporal Integrity Validation** confirms that no future information influences any past observation at any pipeline stage. These are the most scientifically critical validations in this document. A leakage failure invalidates the entire experimental comparison.

**Model Validation** confirms that the architecture behaves as specified — correct tensor shapes, zero fusion parameters, independent branch training, fixed projection matrix.

**Experiment Validation** confirms that the experimental protocol was executed correctly — identical evaluation across conditions, correct statistical comparisons, correct seed application.

## 3. Validation Specifications

### 3.1 Data Validation

**V-DATA-001**
**Objective:** Confirm that each raw timeframe file contains the expected number of candles covering the full study period without critical gaps.

**Verification Method:** After acquisition, count rows per timeframe and verify the date range of the first and last timestamp. Compute the aggregate missing-candle ratio, and log (as a warning) any single gap between consecutive timestamps that exceeds one candle duration by more than one unit.

**Pass Criteria:** Row counts are within 5% of expected values (15m ≈ 140,256; 1h ≈ 35,064; 4h ≈ 8,766; 1d ≈ 1,461). First timestamp ≤ 2020-01-01 00:00 UTC. Last timestamp ≥ the timeframe's last candle on the end date (candles are open-time-labeled: `(2023-12-31 + 1 day) − candle_duration` → 15m: 23:45, 1h: 23:00, 4h: 20:00, 1d: 00:00 UTC). *(Amended 2026-07-03: the earlier fixed "≥ 23:00 UTC" was hourly-anchored and wrongly rejected correctly-covered 4h/1d data — a bug masked by the M1 end-overshoot until real data was trimmed. Now timeframe-aware; see ADR-022 and CHECKPOINT_LATEST.md Sesi 9.)* **Aggregate** missing-candle ratio ≤ 5% (hard gate). *(Amended by ADR-022, 2026-07-03: the earlier hard criterion "no single gap exceeds one candle duration by more than one unit" is downgraded to a logged WARNING — real multi-hour exchange outages, e.g. the 2020-02-19 ~6h Binance outage, are legitimate market conditions, not data-quality defects, and are reindexed by M3 forward-fill; the aggregate 5% gap-ratio remains the pass/fail gate. See docs/ADR-022_single_gap_warning_not_fatal.md.)*

---

**V-DATA-002**
**Objective:** Confirm OHLCV integrity constraints hold across all timeframes.

**Verification Method:** For each row in each timeframe, verify that high ≥ open, high ≥ close, low ≤ open, low ≤ close, high ≥ low, all prices are positive, and volume is non-negative. Verify no NaN or Inf values exist in any column.

**Pass Criteria:** Zero violations across all timeframes.

---

**V-DATA-003**
**Objective:** Confirm the aligned master DataFrame has the correct schema and row count after temporal alignment.

**Verification Method:** After alignment, verify the master DataFrame contains exactly 21 columns as specified in DS-02 Stage 2, that the timestamp column is strictly monotonic and UTC-localized, and that no NaN values exist except in documented boundary rows.

**Pass Criteria:** 21 columns present with correct names and dtypes. Row count ≈ 35,064. Zero NaN values outside documented boundary rows.

---

**V-DATA-004**
**Objective:** Confirm the feature matrix has the correct schema and row count after feature engineering.

**Verification Method:** After feature engineering and NaN row removal, verify the feature DataFrame contains 29 columns as specified in DS-02 Stage 3, that the first timestamp is **2020-01-01 19:00:00 UTC** (reflecting the 19-row NaN drop from the 2020-01-01 00:00 UTC start), and that no NaN or Inf values remain. *(Corrected in v1.1 — previously misstated as "2020-01-19 00:00 UTC"; see document changelog and DS-02 v1.1 changelog.)*

**Pass Criteria:** 29 columns present. Row count ≈ 35,045. First timestamp = 2020-01-01 19:00:00 UTC. Zero NaN or Inf values.

---

**V-DATA-005**
**Objective:** Confirm the temporal split boundary is correct and no overlap exists between train and test sets.

**Verification Method:** Verify that the maximum timestamp in the train set is 2022-12-31 23:00:00 UTC and the minimum timestamp in the test set is 2023-01-01 00:00:00 UTC. Verify that the union of train and test timestamps contains no duplicates.

**Pass Criteria:** Train maximum timestamp = 2022-12-31 23:00:00 UTC exactly. Test minimum timestamp = 2023-01-01 00:00:00 UTC exactly. Zero duplicate timestamps across the two sets.

### 3.2 Temporal Integrity Validation

These validations correspond directly to the leakage checkpoints LC-1 through LC-4 defined in DS-02.

**V-LEAK-001** *(corresponds to LC-2)*
**Objective:** Confirm that forward-fill alignment for 4h and 1d timeframes does not introduce look-ahead information.

**Verification Method:** For a sample of timestamps T in the aligned master DataFrame, verify that the 4h and 1d values assigned to T reflect only the candle that opened at or before T, never a candle that opens after T. Select at least one timestamp from the start of each year in the study period.

**Pass Criteria:** For every sampled timestamp T, the assigned 4h value corresponds to the 4h candle whose open timestamp is the largest value ≤ T. Same condition holds for 1d. Zero look-ahead assignments found.

---

**V-LEAK-002** *(corresponds to LC-3)*
**Objective:** Confirm that no future price information crosses the train/test split boundary.

**Verification Method:** Verify that the rolling statistics used for volume_zscore computation in the feature matrix are derived solely from rows within the same split. Specifically, confirm that the first row of the test feature set does not incorporate any volume values from the training period in its rolling computation.

**Pass Criteria:** The volume_zscore value for the first row of the test feature set is computed using only values from the test period or is NaN. No training-period values contribute to any test-period rolling statistic.

---

**V-LEAK-003** *(corresponds to LC-4)*
**Objective:** Confirm that train windows and test windows are generated exclusively from their respective feature splits and that no window spans the split boundary.

**Verification Method:** Verify that the earliest timestamp in any train window is within the train period, and the latest timestamp in any train window does not exceed 2022-12-31 23:00 UTC. Verify that test windows are generated only from test_features.parquet. Identify and record the number of test windows whose earliest timestamp falls before 2023-01-01 00:00 UTC (expected: up to 47).

**Pass Criteria:** Zero train windows contain timestamps from the test period. All test windows are generated from test feature data only. The count of test windows with any training-period timestamp is documented and matches the expected value of at most 47.

---

**V-LEAK-004**
**Objective:** Confirm that per-window z-score normalization uses only values within each window and does not incorporate any external statistics.

**Verification Method:** For a sample of windows from both train and test sets, manually compute the mean and standard deviation from the raw window values and verify they match the normalization statistics used during processing. Confirm that no global mean or standard deviation computed from the full dataset is used.

**Pass Criteria:** For every sampled window, the normalization mean and standard deviation match values computed solely from that window's 48 rows. No global statistics are applied.

### 3.3 Model Validation

**V-MODEL-001**
**Objective:** Confirm that each branch encoder produces embeddings of the correct shape.

**Verification Method:** Pass a batch of windows through each trained branch encoder and verify the output tensor shape. Test with at least two different batch sizes to confirm the batch dimension is handled correctly.

**Pass Criteria:** For any batch of size B, each branch encoder produces an output tensor of shape [B, 64].

---

**V-MODEL-002**
**Objective:** Confirm that the fusion module produces the correct output shape for every experimental condition.

**Verification Method:** For each of the seven conditions (1TF, 2TF, 3TF, 4TF, BL-15m, BL-4h, BL-1d), pass the appropriate branch embeddings through the fusion module and verify the output shape.

**Pass Criteria:** For every condition and any batch size B, the fusion module produces an output tensor of shape [B, 256].

---

**V-MODEL-003**
**Objective:** Confirm that the fusion projection matrix is fixed and contains zero trainable parameters.

**Verification Method:** Inspect the fusion module's parameter count. Verify that the projection matrix has requires_grad = False. Verify that the projection matrix values are identical before and after a simulated training step that passes gradients through the branch encoders.

**Pass Criteria:** Fusion module trainable parameter count = 0. Projection matrix values unchanged after gradient computation.

---

**V-MODEL-004**
**Objective:** Confirm that the projection matrix is deterministic given the same seed.

**Verification Method:** Generate the projection matrix twice using the same PROJECTION_SEED and condition. Compare the two matrices element-wise.

**Pass Criteria:** Both matrices are identical. Maximum absolute difference = 0.

---

**V-MODEL-005**
**Objective:** Confirm that branch encoders trained independently produce different representations than they would if initialized with a shared seed but trained on different data.

**Verification Method:** After training all four branches, verify that the embedding distributions of the 1h branch and the 4h branch are not identical. Compute the mean cosine similarity between corresponding embeddings from different branches on the same windows.

**Pass Criteria:** Mean cosine similarity between embeddings from any two different branches is less than 1.0, confirming that branches have learned distinct representations. This is an existence check, not a quality threshold.

---

### 3.4 Experiment Validation

**V-EXP-001**
**Objective:** Confirm that all experimental conditions use the same locked HDBSCAN parameters in the primary experiment.

**Verification Method:** Inspect the logged HDBSCAN parameters for all primary conditions (1TF, 2TF, 3TF, 4TF) across all five seeds. Verify that min_cluster_size and min_samples are identical for all conditions within each seed, and that these values match the locked parameters selected from the 1TF grid search.

**Pass Criteria:** min_cluster_size and min_samples are identical across all primary conditions for every seed.

---

**V-EXP-002**
**Objective:** Confirm that all five seeds were applied and that each seed produced a distinct set of embeddings.

**Verification Method:** Verify that five result files exist per condition. For the same condition, compute the mean absolute difference between embeddings produced by different seeds.

**Pass Criteria:** Five result sets exist per condition. Embeddings from different seeds are not identical (mean absolute difference > 0).

---

**V-EXP-003**
**Objective:** Confirm that the Wilcoxon signed-rank tests use correctly paired seed results.

**Verification Method:** Verify that for each comparison (2TF vs 1TF, 3TF vs 1TF, 4TF vs 1TF), the Silhouette Score at seed index i for the multi-resolution condition is compared against the Silhouette Score at seed index i for 1TF. Verify that Holm-Bonferroni correction is applied across all three p-values.

**Pass Criteria:** Each comparison uses five pairs of scores matched by seed index. Three corrected p-values are reported. The correction procedure is documented in the results.

---

**V-EXP-004**
**Objective:** Confirm that external baselines (HMM, KM-PCA) were run under the same five-seed protocol as the primary conditions.

**Verification Method:** Verify that five result sets exist for each external baseline, one per seed, and that the random seed was applied before each run.

**Pass Criteria:** Five result sets exist for HMM and five for KM-PCA. Seed values match [42, 123, 456, 789, 1024].

---

**V-EXP-005**
**Objective:** Confirm that geometric metrics were computed on noise-excluded embeddings for all conditions.

**Verification Method:** For each condition and seed, verify that the embeddings and labels passed to Silhouette, DBI, and CH Index computations contain no points with label = -1.

**Pass Criteria:** Zero noise-labeled points present in any metric computation input.

## 4. Research Invariant Validations

One validation item per INV defined in DS-03 Section 1.

**V-INV-001** *(INV-001 — Single Independent Variable)*
**Objective:** Confirm that all controlled variables are identical across conditions.

**Verification Method:** Compare the configuration snapshots of all experimental runs. Verify that every field in the Controlled Variables table (DS-03 Section 3) is identical across all conditions and all seeds. The only fields that may differ are active_branches and condition label.

**Pass Criteria:** All controlled variable fields are identical across all condition configurations.

---

**V-INV-002** *(INV-002 — Fixed Output Dimension)*
**Objective:** Confirm that every condition produces 256-dimensional embeddings.

**Verification Method:** Inspect the shape of the saved embedding arrays for all conditions and seeds.

**Pass Criteria:** All embedding arrays have shape [N, 256] where N is the number of windows. No condition produces a different second dimension.

---

**V-INV-003** *(INV-003 — Zero Learnable Fusion Parameters)*
**Objective:** Confirm the fusion step is parameter-free. Covered by V-MODEL-003.

**Pass Criteria:** See V-MODEL-003.

---

**V-INV-004** *(INV-004 — Independent Branch Training)*
**Objective:** Confirm that no branch encoder receives gradient signal from another branch during training.

**Verification Method:** Inspect the training implementation and verify that each branch is trained in a separate training loop with its own optimizer, operating on its own timeframe data. Verify that no shared loss function combines outputs from multiple branches.

**Pass Criteria:** Each branch has an independent optimizer instance. No loss function references more than one branch encoder's output simultaneously.

---

**V-INV-005** *(INV-005 — Identical Evaluation Protocol)*
**Objective:** Confirm that the same evaluation procedure is applied to all conditions. Covered by V-EXP-001 and V-EXP-005.

**Pass Criteria:** See V-EXP-001 and V-EXP-005.

---

**V-INV-006** *(INV-006 — Temporal Integrity)*
**Objective:** Confirm that no future information enters any pipeline stage. Covered collectively by V-LEAK-001 through V-LEAK-004.

**Pass Criteria:** See V-LEAK-001 through V-LEAK-004.

---

**V-INV-007** *(INV-007 — Deterministic and Reproducible Execution)*
**Objective:** Confirm that re-running the pipeline with the same seed and configuration produces the same embeddings and cluster labels.

**Verification Method:** Run the full pipeline twice for one condition and one seed. Compare the resulting embedding arrays and cluster label arrays element-wise.

**Pass Criteria:** Embedding arrays are identical across both runs. Cluster label arrays are identical across both runs. Any known non-deterministic GPU operations are documented.

---

**V-INV-008** *(INV-008 — No Hidden Optimization)*
**Objective:** Confirm that no undocumented learnable components exist in the pipeline.

**Verification Method:** Inspect the total trainable parameter count of the full pipeline at inference time (after all branches are trained). Verify it matches the expected count: four independent TS2Vec encoders with no additional components.

**Pass Criteria:** Total trainable parameter count at inference equals the sum of the four branch encoder parameter counts. Fusion module and all post-encoder components contribute zero trainable parameters.
