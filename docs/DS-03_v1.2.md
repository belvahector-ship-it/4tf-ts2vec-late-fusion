## Purpose

This document answers one question: **how should this research be conducted?**

It defines the experimental rules, protocol, and scientific constraints that every implementation must follow. It does not describe how any module is implemented — that lives in the source code. It does not describe how data flows through the pipeline — that lives in DS-02.

Any deviation from this protocol must be explicitly documented and justified. Undocumented deviations invalidate direct comparison with the reported experimental results.

---

## 1. Research Invariants

These are the non-negotiable rules of the experiment. They exist to ensure that temporal resolution is the only variable that differs between experimental conditions. Any implementation decision that would violate an invariant must be escalated and resolved before proceeding.

**INV-001 — Single Independent Variable**
Only the set of active temporal resolutions may differ between experimental conditions. Everything else — architecture, training procedure, hyperparameters, normalization, clustering algorithm, evaluation protocol, random seeds — must be identical across all conditions.

**INV-002 — Fixed Output Dimension**
Every experimental condition must produce a 256-dimensional embedding vector. No condition may produce a different dimensionality. This eliminates embedding dimension as a confounding factor in clustering and metric comparisons.

**INV-003 — Zero Learnable Fusion Parameters**
The fusion step introduces no trainable parameters. The projection from concatenated branch embeddings to 256 dimensions uses a fixed, non-trainable, seeded random matrix. No gradient flows through the fusion step during any training phase.

**INV-004 — Independent Branch Training**
Each TS2Vec branch encoder is trained independently on its own timeframe. No branch receives gradient signal from another branch at any point. The trained weights of each branch are fixed before fusion occurs.

**INV-005 — Identical Evaluation Protocol**
The same clustering algorithm, the same hyperparameter selection procedure, the same geometric metrics, the same economic metrics, and the same statistical tests are applied identically to every experimental condition. No condition receives special evaluation treatment.

**INV-006 — Temporal Integrity**
No implementation may allow future information to influence past observations at any pipeline stage. This applies to normalization, feature computation, alignment, window generation, clustering, and evaluation.

**INV-007 — Deterministic and Reproducible Execution**
Given the same configuration and the same random seed, every pipeline stage must produce deterministic and reproducible outputs. Where the execution environment guarantees bit-identical results, that guarantee should be preserved. Where it does not — for example, certain GPU operations — the implementation must document the limitation and apply all available determinism controls provided by the framework.

**INV-008 — No Hidden Optimization**
No module may introduce additional learnable components or optimization objectives beyond what is explicitly specified in this document and DS-01. The experimental comparison must reflect only the effect of temporal resolution, not any undocumented architectural addition.

## 2. Research Hypotheses

**H0 (Null):** There is no statistically significant difference in clustering quality between single-resolution and multi-resolution conditions as measured by Silhouette Score (p ≥ 0.05, Wilcoxon signed-rank test, after multiple-comparison correction).

**H1 (Primary):** Multi-resolution temporal input produces statistically significantly better clustering quality than single-resolution input. Evaluated separately for 2TF vs 1TF, 3TF vs 1TF, and 4TF vs 1TF (one-tailed Wilcoxon signed-rank tests, α = 0.05 after Holm-Bonferroni correction).

**H2 (Exploratory):** The relationship between the number of active temporal resolutions and clustering quality is non-monotonic. Adding more resolutions beyond a certain point may not improve — and may reduce — clustering quality. This hypothesis is exploratory and does not influence the primary statistical conclusions.

## 5. Training Protocol

### Branch Training

Each of the four branch encoders (15m, 1h, 4h, 1d) is trained exactly once per seed. Trained weights are reused across all conditions that include that branch.

The scientific framing is: *each branch independently learns a single-resolution representation. The multi-resolution conditions evaluate the effect of fusing independently learned single-resolution representations — not the effect of jointly learning multi-resolution representations.*

Training uses the original TS2Vec training protocol without modification. The TS2Vec repository is pinned to a specific commit recorded in requirements.txt and every checkpoint. No TS2Vec source files are modified.

### HDBSCAN Protocol

**Stage 1 — Primary (locked parameters):** Grid search is performed on 1TF training embeddings only. The configuration maximizing Silhouette Score with 2 ≤ k ≤ 8 clusters is locked and applied to all conditions without modification. This is the primary reported result.

**Stage 2 — Sensitivity analysis (optional robustness check):** After completion of the primary experiment, an independent grid search may be performed for each condition. The sensitivity analysis does not affect the primary conclusions. Results are reported separately and clearly labeled if the analysis is conducted.

Grid search space: min_cluster_size ∈ [50, 100, 200], min_samples ∈ [5, 10, 20].

### Protocol B — Strict Test Boundary (Optional Robustness Check)

The primary evaluation (Protocol A) allows the first 47 test windows to contain training-period timestamps, consistent with standard self-supervised learning practice. Primary conclusions rely on Protocol A.

Protocol B excludes these 47 windows and is an optional robustness check. If Protocol B is conducted and its conclusions differ from Protocol A, the difference is reported and discussed explicitly.

## 6. Statistical Protocol

**Seeds:** Five seeds [42, 123, 456, 789, 1024] are applied to all randomness sources for all experimental methods, including HMM and K-Means+PCA. Every run is identified by its condition label and seed.

**Reporting:** All metrics reported as mean ± standard deviation across five seeds.

**Primary comparisons:** Three paired Wilcoxon signed-rank tests are performed:

- 2TF vs 1TF
- 3TF vs 1TF
- 4TF vs 1TF

All tests are one-tailed (alternative = greater), α = 0.05. Because three comparisons are conducted simultaneously, p-values are adjusted using Holm-Bonferroni correction. Effect size reported as rank-biserial correlation r for each comparison.

**Economic validity test:** Kruskal-Wallis test on per-cluster return distributions. Significance threshold p < 0.05.

**Total runs:** 7 primary and secondary conditions × 5 seeds = 35 TS2Vec runs. 2 external baselines × 5 seeds = 10 additional runs. **Total: 45 runs.**

## 7. Acceptance Criteria

A single experimental run is valid if:

- The protocol was executed with the specified configuration and random seed
- Embeddings were successfully generated for the condition
- Clustering completed and labels were assigned
- All predefined evaluation metrics were computed

An experiment across all conditions is valid if:

- All runs satisfy the single-run criteria above
- The same locked HDBSCAN parameters were applied identically to all primary conditions
- The Wilcoxon tests were computed on paired seed results — the same seed index compared across conditions
- No condition received any evaluation treatment not applied to all other conditions

## 9. Reproducibility Requirements

Any researcher must be able to reproduce all results by:

- Cloning the repository at the published release
- Installing the provided dependencies
- Running the acquisition and experiment pipeline with the provided configuration files
- Inspecting the outputs directory for all metrics, figures, and statistical test results

All configuration files, random seeds, and the pinned TS2Vec commit must be included in the repository. No manual steps or undocumented dependencies are permitted.
