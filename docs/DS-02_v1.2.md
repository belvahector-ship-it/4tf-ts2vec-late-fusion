## Multi-Resolution Temporal Encoding for Self-Supervised Cryptocurrency Market State Discovery

> **Document version:** 1.2
> **Status:** Approved
> **Date:** 2026-07
> **Author:** Research Software Engineer (RSE)
> **Depends on:** DS-01 (ADR-001 through ADR-020), research_blueprint.md v1.0

> **Changelog v1.1 → v1.2 (structural design clarification, no research decision changed):**
> - **Stage 5 "Window Extraction" and "Output Tensor Shapes"** corrected: windows are generated from the **full feature matrix** (`btc_features_all.parquet`, 35,045 rows, M4 output) and THEN categorized as train/test based on each window's **anchor timestamp** (the last row of the window) — NOT generated separately from the already-split `train_features.parquet`/`test_features.parquet` files. This was a structural inconsistency (not a simple arithmetic error like the v1.0→v1.1 corrections): the v1.1 text simultaneously claimed windows come "only" from the pre-split files AND that the first test window's earliest timestamp reaches back into the training period — these two claims cannot both be true. Cross-checking against DS-03 v1.2 (§3 CF-002, §5 Protocol B) and DS-04 v1.1 (V-LEAK-003, which expects and requires documenting up to 47 overlap windows as a PASS criterion, not a failure) shows 4 independent, mutually-consistent references supporting the "window-then-split-by-anchor" design, versus only the Stage 5 tensor-shape table supporting the "split-then-window" design. The tensor-shape table was corrected to match the 4-way consensus; see AUDIT_LC4_ADDENDUM.md for the full arithmetic proof (N_test_windows = 8,760 − 48 + 1 = 8,713 under split-then-window, versus exactly 8,760 windows with exactly 47 of them showing boundary overlap under window-then-split — only the latter is consistent with DS-03/DS-04).
> - **`N_test_windows` corrected: 8,760 − 48 + 1 ≈ 8,713 → exactly 8,760** (equal to the number of anchor timestamps in the test period, since windowing is now performed before the train/test categorization, not after row-count truncation of an already-split file).
> - Added explicit note that `train_features.parquet` / `test_features.parquet` (M5 output, DS-02 v1.1 Stage 4, unchanged) are retained as audit/economic-validity artifacts (used in Stage 9 evaluation to join cluster labels back to OHLCV/returns) but are **no longer the direct input to Stage 5 window generation** — Stage 5 now reads Stage 3's full feature matrix directly.
> - This changes the module dependency structure: M6 (Window Generation) now depends only on M4 (Feature Engineering), not on M5 (Temporal Split) — see IMP-01 v1.3 changelog for the corresponding roadmap update.

> **Changelog v1.0 → v1.1 (internal consistency fix, no research decision changed):**
> - **Stage 4 "Expected Sizes" table and footnote** corrected: the claimed first-timestamp-after-NaN-drop "2020-01-19 00:00 UTC" → **"2020-01-01 19:00:00 UTC"**. The pipeline operates at 1-hour resolution starting 2020-01-01 00:00 UTC; Stage 3 "NaN Handling" (unchanged, already correct) states "rows 0–18 are dropped, leaving 35,064 − 19 = 35,045 rows" — dropping 19 hourly rows from 2020-01-01 00:00 UTC lands on 2020-01-01 19:00 UTC, not 2020-01-19 00:00 UTC (which is 432 hours / 18 days later, and was an arithmetic/transcription error in v1.0). The row counts **35,045** (Stage 3), **~26,269** (Stage 4 train), **~26,222** (Stage 5 N_train_windows), and **29 columns** were already correct and are unchanged.

## Stage 1: Data Validation

### Purpose
Detect and abort on any data integrity issue before it propagates through the pipeline.

### Validation Checks (applied to each timeframe independently)

| Check | Condition | Action on failure |
|---|---|---|
| **Timestamp monotonicity** | `timestamps == timestamps.sort_values()` | Abort with sorted index report |
| **Duplicate timestamps** | `timestamps.nunique() == len(timestamps)` | Abort with duplicate rows listed |
| **Missing candle detection** | Gap between consecutive timestamps equals exactly 1 timeframe unit | Log warning for each gap; abort if total missing > 5% |
| **OHLC validity** | `high >= open`, `high >= close`, `low <= open`, `low <= close`, `high >= low` | Abort with violating rows listed |
| **Positive prices** | `open > 0`, `high > 0`, `low > 0`, `close > 0` | Abort |
| **Non-negative volume** | `volume >= 0` | Abort |
| **NaN check** | No NaN in any column | Abort with column-level NaN counts |
| **Inf check** | No Inf/-Inf in any numeric column | Abort |
| **Timezone consistency** | All timestamps are UTC | Abort if mixed or naive |
| **Date range coverage** | First timestamp ≤ 2020-01-01 00:00 UTC, Last timestamp ≥ 2023-12-31 23:00 UTC | Abort |

### Leakage Checkpoint 1
> **LC-1:** No imputation, interpolation, or forward-fill is performed in Stage 1. Gaps are logged only. This ensures that Stage 2 (alignment) is the sole location where upsampling occurs, making leakage auditing unambiguous.

### Output
Validated raw parquet files (same files, same schema, validation report appended to manifest).

## Stage 2: Temporal Alignment

### Purpose
Produce a single aligned master DataFrame where every 1h timestamp from 2020-01-01 00:00 UTC to 2023-12-31 23:00 UTC has exactly one row, containing OHLCV data from all four timeframes.

### Alignment Operations

#### 2a. 15m → 1h Aggregation

Group every 4 consecutive 15m candles into one 1h candle using standard OHLCV aggregation:

```
open   = first(open)    # opening price of first 15m candle
high   = max(high)      # highest high across 4 candles
low    = min(low)       # lowest low across 4 candles
close  = last(close)    # closing price of last 15m candle
volume = sum(volume)    # total volume across 4 candles
timestamp = floor(timestamp, '1H')  # 1h period start
```

Output shape: same as 1h raw (±35,064 rows).

#### 2b. 1h → 1h (Identity)

The 1h timeframe is used directly as the alignment anchor. No transformation needed.

#### 2c. 4h → 1h Forward-Fill

Each 4h candle is broadcast to the next 4 consecutive 1h timestamps using forward-fill:

```
For each 4h candle at timestamp T:
    Assign its OHLCV values to 1h timestamps: T, T+1h, T+2h, T+3h
```

Prefix columns with `_4h` to distinguish from 1h values in the master DataFrame.

#### 2d. 1d → 1h Forward-Fill

Each 1d candle is broadcast to the next 24 consecutive 1h timestamps:

```
For each 1d candle at timestamp T (midnight UTC):
    Assign its OHLCV values to 1h timestamps: T, T+1h, ..., T+23h
```

Prefix columns with `_1d` to distinguish.

### Leakage Checkpoint 2
> **LC-2:** Forward-fill uses only the current candle's values, never future candle values. At timestamp T, the 4h branch value reflects the 4h candle that opened at or before T, not after T. This must be verified by unit test (DS-06 Test ID T-003).

### Boundary Handling

- **Start of dataset:** The first 4h candle starts at 2020-01-01 00:00 UTC. All four 1h slots (00:00, 01:00, 02:00, 03:00) are filled from this candle. No boundary issue.
- **End of dataset:** The last 1d candle at 2023-12-31 00:00 UTC fills all 24 hours of December 31. No boundary issue.
- **Missing candles:** If a raw candle is missing (gap detected in Stage 1), its forward-filled value carries the last known value. This is flagged in the manifest as a partial data zone.

## Stage 3: Feature Engineering

### Purpose
Compute the 7 OHLCV-derived features (ADR-015) for each timeframe column group independently.

### Input
`data/interim/btc_aligned_1h.parquet` — 35,064 rows × 21 columns

### Feature Computation (per timeframe suffix `_tf`)

| Feature | Formula | Notes |
|---|---|---|
| `open_return_tf` | `(open_tf - prev_close_tf) / prev_close_tf` | NaN for first row (no prev_close) |
| `high_return_tf` | `(high_tf - open_tf) / open_tf` | |
| `low_return_tf` | `(low_tf - open_tf) / open_tf` | |
| `close_return_tf` | `(close_tf - open_tf) / open_tf` | |
| `volume_zscore_tf` | `(volume_tf - rolling_mean20_tf) / rolling_std20_tf` | NaN for first 19 rows; rolling window = 20 in native resolution of each tf |
| `hl_range_tf` | `(high_tf - low_tf) / open_tf` | |
| `body_ratio_tf` | `abs(close_tf - open_tf) / (high_tf - low_tf + 1e-8)` | +1e-8 prevents division by zero on doji candles |

### Rolling Window Clarification for `volume_zscore`
The rolling window of 20 is applied to each timeframe column independently over the aligned 1h-anchor rows. This means:
- For `volume_1h`: rolling over 20 consecutive 1h timestamps = 20 hours
- For `volume_15m`: rolling over 20 consecutive 1h timestamps, but the underlying 15m data was aggregated, so the effective window is 20 hours
- For `volume_4h`: rolling over 20 consecutive 1h timestamps, but each value is forward-filled from a 4h candle. The effective window is at most 5 distinct 4h candles (80 hours)
- For `volume_1d`: rolling over 20 consecutive 1h timestamps. The effective window covers at most 1 distinct daily candle

This behavior is acknowledged as a minor, documented confound. It does not affect experimental validity because it is applied identically to all conditions.

### NaN Handling
- First row: `open_return` is NaN (no previous close). **Drop first row.**
- First 19 rows: `volume_zscore` is NaN (insufficient rolling history). Since we also drop the first row for `open_return`, the first 19 rows are dropped.
- Net effect: rows 0–18 are dropped, leaving **35,064 − 19 = 35,045 rows**.

### Output Schema

| Column | dtype |
|---|---|
| `timestamp` | `datetime64[ns, UTC]` |
| `open_return_15m` | `float64` |
| `high_return_15m` | `float64` |
| `low_return_15m` | `float64` |
| `close_return_15m` | `float64` |
| `volume_zscore_15m` | `float64` |
| `hl_range_15m` | `float64` |
| `body_ratio_15m` | `float64` |
| *(repeat × 4 for _1h, _4h, _1d suffixes)* | |

Total columns: 1 + 7 × 4 = **29 columns**  
Total rows: **35,045**

### Output File

```
data/processed/btc_features_all.parquet
```

## Stage 4: Temporal Split

### Purpose
Enforce the walk-forward split (ADR-014). No shuffling. No leakage across boundary.

### Split Boundary
```
Train: timestamp <= 2022-12-31 23:00:00 UTC
Test:  timestamp >= 2023-01-01 00:00:00 UTC
```

### Expected Sizes

| Split | Approx rows | Date range |
|---|---|---|
| Train | ~26,269 | 2020-01-01 19:00 → 2022-12-31 23:00 UTC (after NaN drop) |
| Test | ~8,760 | 2023-01-01 00:00 → 2023-12-31 23:00 UTC |

*Train start is 2020-01-01 19:00 UTC due to the NaN drop of the first 19 rows (hours) from 2020-01-01 00:00 UTC — see Stage 3 "NaN Handling" (35,064 − 19 = 35,045 rows). Corrected in v1.1: previously misstated as "2020-01-19"; see document changelog.*

### Leakage Checkpoint 3
> **LC-3:** The split is performed on feature-engineered data. The rolling mean/std for `volume_zscore` near the split boundary (rows around 2022-12-31 → 2023-01-01) uses data from the 20-row look-back. Since the rolling window is 20 hours and the boundary is a clean year boundary, the last 19 rows of training data influence the first row of test data's rolling volume_zscore. This is documented as follows: the per-window z-score normalization (Stage 5) overrides this effect at inference time, because each W=48 window is normalized using only its own 48 values. The feature-level `volume_zscore_tf` is computed over the rolling 20-candle window and is part of the feature representation, but the per-window normalization in Stage 5 re-centers the entire feature matrix. The interaction is acknowledged but does not constitute leakage: no future price information crosses the boundary.

### Output Files

```
data/processed/
├── train_features.parquet   # ~26,269 rows × 29 columns
└── test_features.parquet    # ~8,760 rows × 29 columns
```

## Stage 5: Window Generation and Normalization

### Purpose
Convert the feature matrix into 3D tensors of sliding windows with per-window z-score normalization, ready for TS2Vec DataLoader consumption.

### Input

`data/processed/btc_features_all.parquet` — 35,045 rows × 29 columns (M4 output, the FULL feature matrix, not the train/test split files). *(Corrected in v1.2 — see document changelog. Window generation reads directly from M4's output; the train/test categorization happens AFTER windowing, based on each window's anchor timestamp — see "Train/Test Categorization" below.)*

### Window Extraction (per timeframe)

For each timeframe `tf ∈ {15m, 1h, 4h, 1d}`, extract a sub-DataFrame of 7 columns from the FULL feature matrix:
```
cols_tf = [open_return_tf, high_return_tf, low_return_tf,
           close_return_tf, volume_zscore_tf, hl_range_tf, body_ratio_tf]
```

Apply sliding window with W=48, stride=1, over all 35,045 rows:
```
For i in range(0, N_rows - W + 1, stride=1):
    window[i] = feature_df[cols_tf].iloc[i : i + W]  # shape [48, 7]
    anchor_timestamp[i] = feature_df['timestamp'].iloc[i + W - 1]  # last row of window
```

### Train/Test Categorization (by anchor timestamp)

After windowing, each window is categorized as train or test based on its **anchor timestamp** (the timestamp of its last row — the same anchor used for joining embeddings back to OHLCV in Stage 9), using the identical ADR-014 boundary as Stage 4:

```
if anchor_timestamp <= 2022-12-31 23:00:00 UTC:
    window belongs to TRAIN
elif anchor_timestamp >= 2023-01-01 00:00:00 UTC:
    window belongs to TEST
```

Because a window looks **backward** from its anchor, a test window's *earliest* row can carry a timestamp from the training period — this is the documented, intentional LC-4 overlap (see below), not a bug. A train window's earliest row is always chronologically before its anchor, so a train window can never contain a timestamp from the test period; leakage is structurally one-directional (earlier information may appear in a later window, but never the reverse) and consistent with INV-006.

### Per-Window Z-Score Normalization

Applied to each extracted window independently, using ONLY that window's own 48 rows — regardless of whether some of those rows chronologically belong to the training period (relevant for the first ~47 test windows, see LC-4):
```
For each window w of shape [48, 7]:
    mu  = w.mean(axis=0)       # shape [7]
    sig = w.std(axis=0)        # shape [7]
    w_norm = (w - mu) / (sig + 1e-8)
```

No external or global statistics (from the full dataset, from the train split, or from any other window) are ever used — this holds regardless of which rows a given window spans, so V-LEAK-004 is satisfied identically for boundary-overlapping and non-overlapping windows alike.

### Output Tensor Shapes

| File | Shape | dtype |
|---|---|---|
| `train_windows_15m.npy` | `[N_train_windows, 48, 7]` | float32 |
| `train_windows_1h.npy` | `[N_train_windows, 48, 7]` | float32 |
| `train_windows_4h.npy` | `[N_train_windows, 48, 7]` | float32 |
| `train_windows_1d.npy` | `[N_train_windows, 48, 7]` | float32 |
| `test_windows_15m.npy` | `[N_test_windows, 48, 7]` | float32 |
| `test_windows_1h.npy` | `[N_test_windows, 48, 7]` | float32 |
| `test_windows_4h.npy` | `[N_test_windows, 48, 7]` | float32 |
| `test_windows_1d.npy` | `[N_test_windows, 48, 7]` | float32 |

Expected N_train_windows ≈ 26,269 − 48 + 1 ≈ **26,222**
Expected N_test_windows = **8,760** exactly *(corrected in v1.2 — previously misstated as "≈8,713"; see document changelog. Since windows are now generated from the full matrix and categorized by anchor timestamp rather than truncated from an already-split 8,760-row file, N_test_windows equals the number of anchor timestamps falling in the test period, i.e. exactly 8,760, not 8,760 − 47.)*

Also saved: `train_timestamps.npy` and `test_timestamps.npy` — the anchor timestamp of each window, shape `[N_windows]`, dtype int64 (Unix nanoseconds UTC). Used to join embeddings back to OHLCV in Stage 9.

### Leakage Checkpoint 4
> **LC-4:** Window generation uses only `iloc` slicing over the full feature matrix. No future information enters any window — normalization and feature values for row *t* never depend on any row after *t*. Windows are categorized as train or test strictly by their anchor timestamp (the last, most recent row), using the same ADR-014 boundary as Stage 4. Because each window looks backward 47 rows from its anchor, the first window with anchor = 2023-01-01 00:00 UTC has an earliest row at 2023-01-01 00:00 − 47h = 2022-12-30 01:00 UTC — i.e. **the first 47 test windows contain rows from the training period**, though every window's anchor (and therefore its role in evaluation and embedding-to-OHLCV joins) remains strictly within the test period. This is documented as an intentional and unavoidable consequence of the sliding window design (DS-03 §3 CF-002), consistent with standard practice in time series self-supervised learning evaluation. Alternative: test windows are generated only from anchors ≥ 2023-02-17 (ensuring no window looks back before 2023-01-01) — this is Protocol B (DS-03 §5), an optional robustness check that does not affect primary conclusions. Decision: document and keep the overlap for the primary analysis (Protocol A). *(Corrected in v1.2 to remove the internal contradiction present in v1.1, which simultaneously claimed test windows came "only" from the pre-split test_features.parquet file — see document changelog and AUDIT_LC4_ADDENDUM.md.)*

## Stage 6: Branch Encoding

### Purpose
Produce a 64-dim embedding vector for each window, for each timeframe branch, using independently trained TS2Vec encoders.

### Input per branch
`train_windows_{tf}.npy` — shape `[N_train_windows, 48, 7]`

### TS2Vec Encoding

TS2Vec produces a representation for each timestamp within the window. The window-level representation is obtained by max-pooling over the time axis:

```
r_window = max_pool(TS2Vec(window), axis=time)   # shape [64]
```

This follows the TS2Vec paper's protocol for instance-level representation (used for classification).

### Output per branch (4 files for train, 4 for test)

| File | Shape | dtype |
|---|---|---|
| `embeddings_train_15m.npy` | `[N_train_windows, 64]` | float32 |
| `embeddings_train_1h.npy` | `[N_train_windows, 64]` | float32 |
| `embeddings_train_4h.npy` | `[N_train_windows, 64]` | float32 |
| `embeddings_train_1d.npy` | `[N_train_windows, 64]` | float32 |
| `embeddings_test_15m.npy` | `[N_test_windows, 64]` | float32 |
| `embeddings_test_1h.npy` | `[N_test_windows, 64]` | float32 |
| `embeddings_test_4h.npy` | `[N_test_windows, 64]` | float32 |
| `embeddings_test_1d.npy` | `[N_test_windows, 64]` | float32 |

## Stage 7: Fusion

### Purpose
Concatenate active branch embeddings and apply fixed random projection to produce a standardized 256-dim representation for each condition.

### Concatenation (per condition)

| Condition | Active branches | Concat shape | Order |
|---|---|---|---|
| {15m} | [e_15m] | [N, 64] | — |
| {1h} (1TF) | [e_1h] | [N, 64] | — |
| {4h} | [e_4h] | [N, 64] | — |
| {1d} | [e_1d] | [N, 64] | — |
| 2TF | [e_15m, e_1h] | [N, 128] | fine→coarse |
| 3TF | [e_15m, e_1h, e_4h] | [N, 192] | fine→coarse |
| 4TF | [e_15m, e_1h, e_4h, e_1d] | [N, 256] | fine→coarse |

### Fixed Random Projection (ADR-003)

For each condition with concatenated dimension D_in ∈ {64, 128, 192, 256}:

```python
# Generated once, deterministically, never updated
generator = torch.Generator().manual_seed(PROJECTION_SEED)  # PROJECTION_SEED=42
P = torch.randn(256, D_in, generator=generator)
# L2-normalize each row
P = P / P.norm(dim=1, keepdim=True)
P.requires_grad = False

# Apply
z_256 = (P @ z_concat.T).T   # shape [N, 256]
```

Note: For the 4TF condition, D_in = 256 = D_out, so the projection is still applied (it is a random rotation, not identity). This is intentional: all conditions pass through the same type of transformation.

### Output per condition

| File | Shape | dtype |
|---|---|---|
| `embeddings_train_{condition}.npy` | `[N_train_windows, 256]` | float32 |
| `embeddings_test_{condition}.npy` | `[N_test_windows, 256]` | float32 |

## Stage 8: Clustering

### Purpose
Assign market state labels to each observation using HDBSCAN on the 256-dim fused embeddings.

### Protocol (two-stage, ADR-004)

**Stage 8a — Primary (1TF-locked parameters):**

1. Grid search HDBSCAN on `embeddings_train_1tf.npy` (1TF={1h} condition)
2. Select configuration maximizing Silhouette Score with 2 ≤ k ≤ 8
3. Save `hdbscan_params_locked.json`: `{min_cluster_size, min_samples}`
4. Fit HDBSCAN with locked params on each condition's training embeddings
5. Predict labels for test embeddings using `hdbscan.approximate_predict()`

**Stage 8b — Sensitivity (per-condition parameters):**

6. Independently grid search each condition (including 1TF)
7. Save `hdbscan_params_sensitivity.json` with best params per condition
8. Results stored separately under `sensitivity_analysis/`

### Noise Handling
- HDBSCAN assigns label `-1` to noise points
- Noise points are retained in all artifact files with label `-1`
- Noise points are **excluded** from geometric metric computation (Silhouette, DBI, CH)
- Noise percentage is reported as a separate metric per run

### Output Files

### Economic Metrics (computed on test cluster parquet, noise excluded)

| Metric | Computation |
|---|---|
| Mean return per cluster | `close_return_1h.groupby(cluster_label).mean()` |
| Std return per cluster | `close_return_1h.groupby(cluster_label).std()` |
| Skewness per cluster | `close_return_1h.groupby(cluster_label).skew()` |
| Kruskal-Wallis test | `scipy.stats.kruskal(*[returns per cluster])` |

### Statistical Test (across 5 seeds)

```
Wilcoxon signed-rank test:
  x = [Silhouette_bestTF_seed_i for i in 1..5]
  y = [Silhouette_1TF_seed_i    for i in 1..5]
  stat, p = scipy.stats.wilcoxon(x, y, alternative='greater')
```
